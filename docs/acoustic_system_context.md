# Акустична розвідувальна система — Контекст проекту

> Документ для відновлення контексту розробки. Вставити на початку нової сесії з Claude.

---

## 1. Концепція системи

**Призначення:** Акустична розвідка для спостережного пункту (СП) у польових умовах. Визначення джерела, напрямку і характеру звуку в реальному часі.

**Принцип роботи:** Мережа автономних акустичних станцій збирає звук, обчислює TDOA (різниця часу надходження між мікрофонами), передає метадані на СП. Сервер тріангулює точну позицію джерела звуку і класифікує його (людина / тварина / транспорт).

**Радіус покриття:** До 3-5 км від сервера. Відстань між станціями — до 500м.

**Час реакції:** Одиниці секунд — прийнятно.

---

## 2. Архітектура

**Топологія мережі:** Mesh — кожна станція є і сенсором, і ретранслятором. Немає єдиної точки відмови. При виході станції з ладу мережа перебудовується автоматично.

**Два радіоканали:**
- LoRa 433МГц (FHSS) — метадані подій, керуючі команди, синхронізація. Завжди активний, мінімальна радіопомітність.
- 2.4ГГц nRF24L01 PA+LNA — аудіофрагменти. Вмикається тільки за запитом СП на 15-20 сек.

**Передача аудіо за запитом:** Станція має кільцевий буфер 10-15 сек (pre+post trigger). Сервер отримує метадані від всіх станцій → обирає 1-2 з найкращим SNR → надсилає запит → станція вмикає 2.4ГГц передавач і відправляє стиснений фрагмент (Opus, 16кГц mono, ~32кбіт/с).

> **Параметри запису (прототип):** 32 000 Гц / 16-bit. Bandwidth 16 kHz покриває акустичні сигнатури БПЛА (основні гармоніки до 12 кГц). Ring buffer: 32000 × 4 кан. × 2 байт × 15 с = 3.84 МБ (в межах 8 МБ PSRAM N16R8).

**Мікрофон на СП:** СП має власний USB мікрофон як нульова станція. Якщо подія близько (~200-500м) — аудіо є без запиту до станцій.

**3D локалізація:** Тетраедральний масив 4 мікрофонів на кожній станції дає 3D напрямок (азимут + елевація). Розташування: 1 мікрофон вгору, 3 горизонтально під 120°. Сервер тріангулює позицію методами Bearing Intersection + TDOA між станціями.

## 3. Апаратна частина станції

**Процесор:**
- Прототип: ESP32-S3-DevKitC-1 N8R8 (8МБ Flash + 8МБ PSRAM, dual-core 240МГц)
- Фінальна версія: STM32H743 (краща енергетика, 4× незалежних I2S шини)

**Мікрофони:** 4× INMP441 I2S модуль (прототип) або ICS-43434 (фінал, фазова узгодженість ±0.5°). Розташування тетраедральне. Вітрозахисні поролонові насадки обов'язкові.

> **Серійний зразок — збільшення смуги:** перехід на 44 100 Гц / 24-bit з мікрофонами ICS-43434 (частотна характеристика до 20 кГц). Дає повний спектр для класифікації БПЛА та подвоює роздільну здатність GCC-PHAT TDOA. Потребує зовнішнього SDRAM або скорочення ring buffer до 10 с (буфер ~7.9 МБ).

**Синхронізація часу:**
- TCXO ±0.5ppm на кожній станції (дрейф ~1.5мкс/год)
- Калібрування: акустичний маяк при розгортанні + взаємна корекція між станціями по LoRa
- БЕЗ GPS — вразливий до РЕБ

**Живлення:**
- LiFePO4 26650, 2 елементи паралельно = 10А·год @ 3.2В = 32Вт·год
- Середнє споживання ~75мВт → автономність ~18 діб (при 0°C і вище)
- При -20°C: ~10-12 діб. Корпус утеплювати, підігрів при температурі нижче -10°C

**Режими роботи:**
- Очікування (95% часу): мікроконтролер у light sleep, аналоговий компаратор слухає (~56мВт)
- Обробка події (4%): пробудження, 4-канальний запис, GCC-PHAT TDOA, класифікація (~330мВт)
- Передача LoRa (1%): пакет 30 байт кожні кілька секунд (~400мВт × 0.5 сек)

**Корпус:** IP65 пластик мінімум. Кабельні вводи PG7/PG9 для герметизації. Передбачити місце для сонячної панелі (не реалізовано, але конструктивно закладено).

## 4. Базова станція / СП

**Обладнання:**
- Raspberry Pi 4 (4ГБ) або Orange Pi 5 як альтернатива
- RAK2245 Pi HAT (SX1301, 8 каналів LoRa одночасно)
- Окремий ESP32 або nRF52840 для прийому аудіо по 2.4ГГц
- USB мікрофон / аудіоінтерфейс для запису на СП
- SSD 256ГБ для логів і аудіоархіву

**Програмна обробка:**
- Тріангуляція позиції (Bearing Intersection + TDOA між станціями)
- Класифікація звуку нейромережею (людина / тварина / транспорт / невідомо)
- Трекінг переміщення — побудова траєкторії об'єкта
- Фільтрація хибних спрацювань (дощ, вітер, птахи)
- Інтерфейс оператора — карта з позиціями подій

**Живлення СП:** 12В від авто акумулятора або окремий LiFePO4 100Вт·год блок.

## 5. Синхронізація без GPS (захист від РЕБ)

**Проблема:** GPS/GNSS вразливий до спуфінгу і придушення в умовах РЕБ. Втрата GPS одразу ламає TDOA між станціями.

**Рішення — акустичний маяк:** При розгортанні пристрій з відомими координатами генерує короткий імпульс за внутрішнім годинником. Всі станції фіксують час надходження → сервер обчислює відносні затримки → калібрує годинники. GPS потрібен тільки одноразово при встановленні (один пристрій, потім вимикається).

**Рішення — взаємна корекція:** Станції обмінюються мітками часу по LoRa mesh і взаємно коригують дрейф (distributed clock sync). Точність ~1-5мс — достатньо для тріангуляції на дистанціях 300-500м.

**Деградація без синхронізації:** TCXO дрейфує ~1.5мкс/год. Без корекції через 1 годину похибка позиції ~0.5м, через 24 години ~12м. Система продовжує працювати в деградованому режимі.

**Координати станцій:** Вимірюються одним GPS пристроєм послідовно при розгортанні. Вводяться в сервер вручну або передаються одноразово. Після цього GPS вимикається.

## 6. Тестова конфігурація (поточний етап)

**Склад тестової системи:**
- 2× автономні станції (ESP32-S3 + 4×INMP441 + Ra-02 LoRa + nRF24L01 + LiFePO4)
- 1× базова станція — Raspberry Pi 4 + RAK2245 HAT
- Розміщення рівностороннім трикутником ~300м сторона

**Що перевіряємо:**
- TDOA локалізація всередині станції (напрямок на джерело звуку)
- Передача метаданих по LoRa
- Тріангуляція між 2 станціями + СП (3 точки = мінімум для 2D позиції)
- Протокол запиту і передачі аудіофрагменту
- Синхронізація часу між вузлами
- Енергетика і автономність в реальних умовах

**Етапи розробки:**

| Етап | Зміст | Тривалість |
|------|-------|------------|
| 1 | Один вузол на столі — 4-канальний запис + GCC-PHAT TDOA | 1-2 тижні |
| 2 | Зв'язок з базою — LoRa метадані + nRF24 аудіо протокол | 1 тиждень |
| 3 | Два вузли на вулиці — тріангуляція реального джерела, точність | 1-2 тижні |
| 4 | Програмна обробка на СП — карта, класифікація, інтерфейс | TBD |

**Поточний статус:** Закупка компонентів. Далі — Етап 1.

## 7. Ключові прийняті рішення

| Рішення | Обґрунтування |
|---------|---------------|
| Без GPS | Вразливий до РЕБ. Синхронізація — акустичний маяк + TCXO + взаємна корекція |
| Двоканальна радіоархітектура | LoRa для метаданих (завжди), 2.4ГГц тільки для аудіо за запитом |
| Аудіо тільки за запитом | Економія енергії, зниження радіопомітності |
| Pre-trigger буфер | Запис починається ДО події — найцінніша частина звуку зберігається |
| LoRa FHSS | Стрибки по частотах — мінімальна помітність, стійкість до перешкод |
| Температурна поправка | Швидкість звуку змінюється 0.6м/с на градус, при 1-2км це метри похибки |
| LiFePO4 26650 | Краща хімія для вуличних умов і морозу |
| ESP32-S3 для прототипу | STM32H743 — для фінальної версії |

## 8. Структура LoRa пакету (~30 байт)

| Поле | Розмір | Зміст |
|------|--------|-------|
| Station ID | 2 байти | Ідентифікатор станції |
| Timestamp | 8 байт | Мікросекунди, синхронізовано по TCXO |
| TDOA між мікрофонами | 4×4 байти | Попарні затримки між 4 мікрофонами |
| SNR по каналах | 4×1 байт | Рівень сигнал/шум кожного мікрофона |
| Клас події | 1 байт | Людина / тварина / транспорт / невідомо |
| Confidence | 1 байт | Впевненість класифікації (0-100%) |
| Рівень заряду | 1 байт | Відсоток заряду акумулятора |
| Температура | 1 байт | Для корекції швидкості звуку |
